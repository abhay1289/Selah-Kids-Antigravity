# Antigravity Agent — Selah Kids Payload CMS Migration

## Mission

Integrate Payload CMS v3 into the existing Selah Kids Next.js 15 app so the admin panel at `/admin` is fully end-to-end connected with a PostgreSQL database. The public frontend must NOT change — zero visual or behavioral differences for visitors. All content currently hardcoded in `src/data/*.ts` files must be manageable through Payload's admin UI by a non-technical client.

## Critical Rules

1. **DO NOT modify any file under `src/app/[locale]/`** — the public frontend is frozen
2. **DO NOT modify any component under `src/components/`** — the UI is frozen
3. **USE Payload's native `localized: true`** on all translatable fields — NOT hardcoded `titleEn`/`titleEs` pairs
4. **ADD `versions: { drafts: true }`** on BlogPosts collection
5. **ADD `_status: 'published'` filter** in every cms-server.ts query that serves public pages
6. **KEEP offline fallback** — if `DATABASE_URI` is unset, `cms-server.ts` must return data from `src/data/cms-fallbacks.ts` (existing behavior)
7. **USE `bun`** not `npm` for all package operations
8. **BUILD must work in GitHub Actions** (8GB RAM) — never assume VPS build capability

### Step 6: Rewrite `src/lib/cms-server.ts`

Replace all Supabase calls with Payload local API calls. Keep the offline fallback.

**Pattern:**
```ts
import { getPayload } from 'payload';
import config from '@payload-config';
import { INITIAL_BLOG_POSTS } from '@/data/cms-fallbacks';

export async function getBlogPosts(locale: 'en' | 'es' = 'es') {
  if (!process.env.DATABASE_URI) return INITIAL_BLOG_POSTS; // offline fallback

  const payload = await getPayload({ config });
  const { docs } = await payload.find({
    collection: 'blog-posts',
    locale,
    where: { _status: { equals: 'published' } },
    limit: 100,
    sort: '-createdAt',
  });
  return docs;
}

export async function getBlogPostBySlug(slug: string, locale: 'en' | 'es' = 'es') {
  if (!process.env.DATABASE_URI) {
    return INITIAL_BLOG_POSTS.find(p => p.slug === slug || p.slugEs === slug) || null;
  }

  const payload = await getPayload({ config });
  const { docs } = await payload.find({
    collection: 'blog-posts',
    locale,
    where: {
      or: [
        { slug: { equals: slug } },
        { slugEs: { equals: slug } },
      ],
    },
    limit: 1,
  });
  return docs[0] || null;
}
```

Implement the same pattern for:
- `getCharacters(locale)`
- `getEpisodes(locale)`
- `getTeamMembers(locale)`
- `getPageContent(page, locale)`
- `getSiteSettings(locale)`
- `getSeoMetadata(page, locale)`
- `getNavigation(locale)`

## Data Shape Reference

Read these files to understand the exact data structures the frontend expects:
- `src/data/blogPosts.ts` — BlogPost interface
- `src/data/characters.ts` — Character interface
- `src/data/catalog.ts` — Episode interface
- `src/data/team.ts` — TeamMember interface
- `src/data/page-content-home.ts` — PageFieldMap shape (key-value pairs)
- `src/data/chrome-navbar.ts` — NavLink shape
- `src/data/chrome-footer.ts` — FooterLink shape
- `src/data/chrome-seo.ts` — SeoPage shape

The Payload collections must output data that matches these interfaces. The public components consume these shapes via `cms-server.ts` — if the shape changes, components break.
